# Claude-named Run Sessions — Design

**Status:** draft
**Date:** 2026-04-22
**Base branch:** `main` (fbi-ui-redesign has merged; no separate base required)

## Problem

Runs in the UI are currently labeled by a combination of `Run #{id}` and either the branch name or the first line of the prompt. Neither is a good session identifier:

- The prompt is verbose and rarely captures the actual shape of the work.
- The branch name is generated by Claude only at the end of the run (via the supervisor's fallback), so in-flight runs fall back to the prompt preview.
- Neither is editable.

## Goal

Let Claude name the session itself — early (so in-flight runs have a meaningful label) and refined at the end — with the user able to override the name via a UI rename that sticks.

Non-goals: redesigning the runs list, adding a new waiting-for-input state (tracked separately), backfilling titles for historical runs.

## Design

### Data model

Two new columns on `runs`:

| Column | Type | Notes |
|---|---|---|
| `title` | `TEXT` nullable | The displayed session name. Null until Claude emits one. |
| `title_locked` | `INTEGER NOT NULL DEFAULT 0` | Set to 1 when the user renames via UI; blocks further auto-updates. |

Mirror on the `Run` TS interface in `src/shared/types.ts`.

One migration adds both columns via the existing `migrate()` pattern in `src/server/db/index.ts`. No backfill — existing rows render via the fallback chain (`title ?? branch_name ?? first-line(prompt) ?? 'untitled'`), preserving current behavior.

### Write path — two phases

**Phase 1: early title via sidecar file.**

The orchestrator creates a per-run host directory `${runsDir}/{id}/state/` and bind-mounts it to `/fbi-state/` inside the container, writable by the container user. A host-side `TitleWatcher` (sibling of `UsageTailer`, lives in `src/server/orchestrator/`) polls `${runsDir}/{id}/state/session-name` every ~1 second for the lifetime of the run. On each poll, if the file exists, the watcher:

1. Reads the file, trims whitespace, truncates to 80 chars. Skips if empty after trim.
2. Compares against the last-seen value it emitted for this run. If identical, skips (de-dup — no-op when Claude hasn't revised).
3. Calls `RunsRepo.updateTitle(runId, title, { respectLock: true })`. Still a no-op at the repo layer if `title_locked = 1`.
4. Broadcasts a typed `title_updated` WS frame via the existing `RunStreamRegistry` events channel (parallel to usage/rate_limit frames).

Claude may overwrite the sidecar later; the watcher picks up each distinct value, so a revised name propagates without restart.

Supervisor preamble in `src/server/orchestrator/index.ts` gains:

> As soon as you understand the task, write a short name (4–8 words, imperative, no trailing punctuation) describing this session to `/fbi-state/session-name`. You may overwrite it later if your understanding changes. Also include a refined `title` field in the final result JSON.

Error handling: the watcher silently ignores `ENOENT` (file not yet written) but forwards other read errors (e.g., `EACCES`, `EISDIR`) to `onError` so they surface in logs.

**Phase 2: refined title via result JSON.**

Extend `ContainerResult` in `src/server/orchestrator/result.ts` with an optional `title?: string`. `parseResultJson` trims and truncates to 80 chars; omits the field when empty. The orchestrator's finish paths (after `markFinished`) call `updateTitle(runId, parsed.title, { respectLock: true })` and broadcast the typed frame. The lock still prevents stomping a user rename.

**Failure modes.** If Claude never writes the sidecar and never emits `title` in result JSON, `title` stays null and the UI falls back. The watcher is best-effort — failures are logged but never fail the run.

### Read / edit path

**API.** One new route in `src/server/api/runs.ts`:

- `PATCH /api/runs/:id` — body `{ title: string }`. Trims and validates: non-empty, ≤120 chars. Calls `RunsRepo.updateTitle(id, title, { lock: true, respectLock: false })`, broadcasts a title frame, returns the updated run as JSON. 400 on validation failure, 404 on unknown run.

**Client.** `src/web/lib/api.ts` gains `renameRun(id, title)`. `RunsList.tsx` filter adds `title` to its substring search alongside `branch_name` and `prompt`. The command palette's runs group does the same.

**UI changes:**

- `src/web/features/runs/RunRow.tsx` — change the label expression to `run.title ?? run.branch_name ?? run.prompt.split('\n')[0] ?? 'untitled'`. One-line change.
- `src/web/features/runs/RunHeader.tsx` — add inline-editable title alongside the existing `Run #{id}`, Pill, CodeBlock, and button row (Continue/Follow up/Cancel/More). Double-click OR single-click enters edit mode (keyboard-accessible), `Input` primitive seeded from current title, Enter saves (calls `renameRun`), Esc cancels, onBlur commits UNLESS focus moves to a sibling button in the same header (use `relatedTarget` to avoid fighting adjacent clicks). On API failure, keep the editor open so the user can see their unsaved draft. The optional `onRenamed?: (run: Run) => void` callback carries the full Run (so parents can pick up `title_locked` without going stale).

**WS consumption.** The shell registry and `useRunWatcher` already drain typed frames. Add a handler that patches the in-memory run record on `title_updated` so the list and header update live without a refetch.

### Concurrency & ordering

- Early sidecar updates and end-of-run result-JSON updates both go through `RunsRepo.updateTitle` with `respectLock: true`. SQLite's per-statement atomicity is sufficient.
- User rename and orchestrator update can race. The user's PATCH uses `respectLock: false, lock: true`. Any orchestrator update arriving after the lock is a no-op. Any arriving before the lock is overwritten by the user's PATCH (expected: the user's choice wins).
- The `title_updated` frame always reflects post-write DB state, so the UI never sees a stale transient.

### Lessons folded in from earlier design iteration

The following refinements were identified during a prior implementation pass and are baked into this spec from the start:

1. The `RunsRepo.updateTitle` + events-publish sequence appears at multiple orchestrator call sites. It's a single private helper (`publishTitleUpdate(runId, title)`) on the orchestrator class from day one.
2. `TitleWatcher` narrows its inner catch to `ENOENT` specifically; any other errno propagates to `onError`.
3. `RunHeader.onBlur` uses `relatedTarget` to detect in-header focus transitions (so clicking Cancel while editing doesn't fire both commit AND cancel).
4. `RunHeader` fires `onRenamed?.(updatedRun)` — full Run, not just the title string.
5. `RunHeader` supports both single-click and double-click to enter edit mode (keyboard users activate the button via Enter/Space).
6. On API failure in `RunHeader.commit`, the editor stays open so the user sees their draft.

## Testing

**Unit tests (no Docker):**

- `RunsRepo.updateTitle` — happy path, `respectLock` no-op when locked, length/empty validation.
- `TitleWatcher` — missing file silent, first fire trimmed+truncated, de-dup, revision, empty skipped, non-ENOENT errors surface via `onError`.
- `parseResultJson` — accepts `title`, absent `title` still parses, empty/whitespace omitted, oversize trimmed.
- `PATCH /api/runs/:id` — trim / length / empty handling, 404 on unknown run, returns updated run body.
- `RunRow` — fallback chain renders `title` preferentially, falls back through branch → prompt → `untitled`.

**Integration (Docker-gated, same pattern as existing orchestrator tests):**

- Stub Claude container that writes to `/fbi-state/session-name` mid-run and emits a result JSON with `title`. Assert DB has the final title, `title_locked = 0`, and the WS channel received two `title_updated` frames.
- Same setup, but a middle step hits `PATCH /api/runs/:id`. Assert the final title is the user's value, `title_locked = 1`, and the end-of-run title update is a no-op.

**Manual:** Start `scripts/dev.sh`, create a new run, observe that the sidebar label updates live as Claude writes the sidecar. Confirm double-click and single-click rename persists across reload; API failure keeps the editor open.

## Rollout

1. One migration adds `title` and `title_locked` columns.
2. No backfill.
3. Feature branch is `feat/claude-session-naming`, based on `main`.
4. No feature flag — the fallback chain makes the feature degrade gracefully for any run that doesn't emit a title.

## Out of scope (follow-up specs)

- `awaiting_input` run state and associated UI / notifications / signaling mechanism.
- Retroactively titling historical runs.
- Title-based deduplication or clustering of similar sessions.
